# CLAUDE.md

cc-telegram — Telegram bot that bridges Telegram Forum topics to Claude Code sessions via tmux windows. Each topic is bound to one tmux window running one Claude Code instance.

Tech stack: Python, python-telegram-bot, tmux, uv.

## Common Commands

```bash
uv run ruff check src/ tests/         # Lint — MUST pass before committing
uv run ruff format src/ tests/        # Format — auto-fix, then verify with --check
uv run pyright src/cctelegram/        # Type check — MUST be 0 errors before committing
uv run pytest -m scenario -q          # Scenario floor — black-box behavior tests at the public Telegram seam
bin/post-wave-check.sh                # Architecture deepening health diff (LoC, brittleness, tool status)
cc-telegram hook --install            # Auto-install the managed Claude Code hooks (SessionStart / PreToolUse / Notification)
```

## Core Design Constraints

- **1 Topic = 1 Window = 1 Session** — all internal routing keyed by tmux window ID (`@0`, `@12`), not window name. Window names kept as display names. Same directory can have multiple windows.
- **Topic-only** — no backward-compat for non-topic mode. No `active_sessions`, no `/list`, no General topic routing.
- **No message truncation** at parse layer — splitting only at send layer (`split_message`, 4096 char limit).
- **MarkdownV2 only** — use `safe_reply`/`safe_edit`/`safe_send` helpers (auto fallback to plain text). Internal queue/UI code calls bot API directly with its own fallback.
- **Hook-based session tracking** — `SessionStart` hook writes `session_map.json`; monitor polls it to detect session changes.
- **Message queue per user** — FIFO ordering, message merging (3800 char limit), tool_use/tool_result pairing.
- **Rate limiting** — `TypingAwareRateLimiter(max_retries=5)` (an `AIORateLimiter` subclass in `rate_limiter.py`) on the Application (30/s global). On restart, the global bucket is pre-filled to avoid burst against Telegram's server-side counter. The subclass exempts `sendChatAction` (typing) from the per-GROUP 20/60s bucket (positive dummy chat_id to the classifier) while keeping the overall limiter + RetryAfter, so multi-topic typing cadence isn't paced by the message budget.
- **Scenario test floor** — `tests/scenarios/*.py` are black-box behavior tests at the public Telegram seam (`@pytest.mark.scenario`). They drive `Update` → real handler stack → fake tmux / fake bot, with no monkeypatch of handler internals in test bodies. Architecture changes must preserve these scenarios green.
- **RouteRuntime is the run-state authority** — `cctelegram.route_runtime` is the sole run-state / context-usage / idle-clear state machine (fed by `cctelegram.transcript_event_adapter`); it owns `RunState`, `ContextUsage`, and `IDLE_CLEAR_DELAY_SECONDS`. Mutations go through `ingest_transcript_event` / `mark_*`; reads come from `route_runtime.snapshot(route)`. Per-route `asyncio.Lock` only; no `register_state_callback` / `register_activity_callback` fan-out (that pattern produced bug c313657 and is precisely what `RouteRuntime` replaced). `message_queue` remains the only sender/editor of status cards; it queries `snapshot.status_card_visible` and writes back via `mark_status_card_published(route, msg_id)` — if it ever needs to mutate `message_queue` internals beyond that, the kill criterion fires (promote Route Outbox). A pane/lifecycle signal may **PROMOTE an active `RUNNING` route** (empty `open_tools`) to `WAITING_ON_USER` via `mark_interactive_pending` (retract via `mark_interactive_cleared`) for the window where Claude Code buffers the interactive `tool_use` (AskUserQuestion / ExitPlanMode) in JSONL — so the digest reads "🔔 Waiting on you" and typing stops while the prompt is live. Promotion fires only from a **pane-confirmed** live picker/plan-approval in `status_polling` (never the session-keyed side file alone — bit-neutral site (c) can't disambiguate a double-`--resume` sibling). A per-window in-memory registry refuses further payloads until `pane_input_row_empty` proves the box is clear (indeterminate keeps it; no auto-Esc — Esc on folder-trust KILLS Claude); a restart wipes it (disclosed). **The registry lives in `tmux_manager`** (beside the post-/exit quarantine it mirrors; `session.py` keeps the four names as the delivery-path vocabulary) because the brake is a property of the PANE, not of a topic binding — **BINDING-LEVEL TEARDOWN MUST NOT CLEAR IT:** `/unbind` DELIBERATELY leaves the window ALIVE and the teardown seams hold no `window_send_lock`, so clearing there let a send already BLOCKED on that lock append to the leftover draft and commit BOTH. The only other release proof is WINDOW DEATH — a CONFIRMED `kill_window` (topic close / `/kill` route through it) or `create_window` under a reused id (tmux ids reset to `@0` on a tmux-server restart). It is armed through ONE seam, INSIDE the send lock, on BOTH paths: the normal `draft_stranded` return AND a `CancelledError` / raise that lands after a write was ATTEMPTED (a `_WriteAttempt` flag set immediately BEFORE the first literal `send_keys` — a cancelled write can still have landed; the exception always RE-RAISES, cancellation is never swallowed). Requires `CC_TELEGRAM_DECISION_CARDS` ON to matter.

**Free-text answers (`CC_TELEGRAM_FREE_TEXT_ANSWERS`, default ON)** — a plain Telegram message (typed or voice, reply-quoted or not) sent at a live **AskUserQuestion single-select** picker is driven into the card's free-text affordance row and committed, instead of being refused. **ExitPlanMode is NOT covered** (owner decision 2026-07-12) — a plan card takes PR-1's refusal. No NEW state file, no NEW hook, no callback lane — it is a pure delivery-path executor (`handlers/free_text.py`) invoked from the aggregator flush, and its occurrence anchor is the EXISTING `PreToolUse(AskUserQuestion)` side file (which the lane therefore REQUIRES; without it, it declines). `config.py` owns the canonical env var; `main._run_bot` seeds the config-free leaf. Version-licensed per `(surface × CC-version)`; an un-characterized CC release silently reverts to PR-1's refusal. See the Core Design Constraints bullet above.

**AFK auto-resolve → late-answer card (`aql:`, Wave A)** — on Claude Code ≥2.1.198 an unanswered AUQ self-resolves at ~60s (undocumented, no knob; the CLI default is never defeated). `bot.handle_new_message`'s AUQ tool_result branch detects it two-factor (`late_answer.is_afk_auto_resolve`: unanchored "No response after Ns" regex + the AUTHORITATIVE entry-level `toolUseResult.answers` non-empty ⇒ False; meta-absent = sentinel-strip → negative wrappers → anchored start) and, instead of deleting the picker, CONVERTS it in one route-lock critical section (snapshot → Phase-1 pop → `forget_ask_tool_input` → ledger release; Phase-2 edit SHIELDED) into "⏰ Claude proceeded after ~60s (no response)." with `aql:<window_id>:<opt>:<token>` buttons (single-question single-select only; multi-Q/multi-select → text-only). A tap delivers the choice as a NORMAL user text message ("Re your earlier question … my answer is … Please course-correct") through the effort.py route-ordering subsequence with the GH #50 pre-commit user-turn stamp (fired inside the gated delivery transaction, immediately before the Enter) — never a picker keystroke; the aqp:/aqt:/pick_token/ledger machinery is byte-untouched. Token registry is in-memory (`handlers/late_answer.py`; restart ⇒ graceful expired modal), single-use per card, invalidated on the next AUQ's resolution / rotation / topic close; taps are guarded by owner + stale-window + freshness (`has_interactive_surface` / live side file ⇒ "a newer prompt is live"). Non-AFK tool_results keep today's teardown byte-identical; ExitPlanMode is OUT of scope (60s behavior unobserved for EPM).

A third hook event — `MessageDisplay` (Bug 2 live prose) — is **NOT** managed by `cc-telegram hook --install` and is **NOT** in `~/.claude/settings.json`. The bot writes its own `md_hook_settings.json` and passes it only to sessions it launches via `claude --settings <file>` (`tmux_manager._compose_launch_command`), scoping the hook to the bot's windows (it merges with the global `SessionStart`/`PreToolUse`). The hook command runs the tiny stdlib `_md_display_appender.py` directly (it must never import the package — the streaming-display path runs hooks with `forceSyncExecution`, so latency matters; `md_capture` benchmarks the appender against a bare interpreter start). The bot logs a one-time startup warning if it cannot write the settings file, and live prose silently falls back to post-resolution JSONL delivery.

### README sync rule

Any change that adds **a hook** (SessionStart / PreToolUse / Stop / SubagentStop / etc.), **an env var** (`CC_TELEGRAM_*` or external config dependency), **a state file or directory** (under `~/.cc-telegram/` or `~/.claude/`), or **a new external config dependency** (launchd plist, log-rotate agent, etc.) MUST update `README.md` in the same PR — touching at minimum the relevant section among "What it does", "Configure", "Install the Claude Code hook", "State files", "Log rotation", or "Repository layout".
